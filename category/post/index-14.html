  1. Pair Programming

    09 December 2015

    Pair programming can be a lot of fun. I am quite a social person and enjoy sitting with someone to solve a problem. The thing with code is that it can be tackled in so many different ways and it takes some level of compromise to work with someone as they might not think in the same way you do.

    This can be quite tiring, and pair programming is tiring, but incredibly useful. You can learn new things in a practical way and learn how to think laterally.

    Typically there is a driver (who handles the tactical/typing part of the coding) and a navigator (who thinks about the overall strategy). These roles of course switch regularly when ping pong programming.

    Some tips that are useful to think about when pair programming that we covered in our Zagaku lesson today were:

    1. Set the ground rules before you start. (If checking phones or emails is alright, when you both agree to take breaks, how long you want to pair etc.)

    2. Decide on the type of pair programming style you would like to adopt. The style we tend to practice the most at 8th light is ping pong. This is where one person writes a test, then the next writes code to make the test pass. Finally, the first person again has to refactor if possible to make the code better.

    3. Ensure you have a comfortable set up, ideally with one screen and one or two keyboards.

    4. Communication is the key factor, make sure you communicate clearly for the duration of your pair programming session.

    Pairing can seem daunting when you are inexperienced, but I think everyone hopefully gets something out of it, so I am going to try and pair daily and recommend you do to!

  2. Black Box Tests

    10 December 2015

    Testing and knowing what to test can be tough when you begin to code. Starting with a degenerate case is a good approach, but sometimes this can lead to testing implementation details that can change as your code evolves. This is where black box testing comes in. Here you only test the public methods that handle the overall functionality and behaviour you want from a piece of code. White box testing is where you test the internals of a class or individual methods that handle small pieces of behaviour that you eventually want to hide as private methods.

    I know I am guilty of writing simple and small tests that test individual functions and I tend not to change these to black box tests even when I know I no longer need them. I am going to keep in mind the advantages of black box testing and try and think of the overall functionality I want to test more going forward.

  3. Retros

    11 December 2015

    Today I spent the day at a client site and helped facilitate a retrospective in which the participants talked about all the things they learnt over the five weeks during which they built their product.

    It was fun and it was great to see them able to demo their product well. It was also interesting that at one point they mentioned that they were concerned with how the final product looked more than the code underlying it. This can happen quite easily when you want to show something to people who do not understand code very well. But to people who know what lies beneath the user interface the structure and quality of the code is equally if not more important.

    I got the impression that the 8th Light effect rubbed off on them, and they all seemed to really get along well and enjoy what they were doing. The fact that it was their final day might have been a contributing factor, but I hope they can carry their enthusiasm with them when they go on to their new projects and new teams.

  4. Vim

    14 December 2015

    When I first started to learn Ruby, I started with Vim, an advanced text editor that was difficult to navigate at first. My mentor sent me the Vim Adventures game, which is an easy introduction into learning how to navigate using Vim. It was really slow to begin with and I always reverted to using my arrow keys, so these were disabled to force me to use shortcuts. It was annoying and slow at first, but after a while I was noticably faster at writing code.

    When I started to learn Java however, I moved on to the beautiful Intellij which I really enjoyed using. I was able to get started pretty quickly and there are so many autocomplete suggestions and eror checkers in Intellij, it really helped me with my syntax and refactoring. But there is a lot that Intellij does for you that you can take for granted. This is why I think my next task was to use gradle to set up my Human vs Human Tic Tac Toe. Luckily one of the other Apprentices, Georgina, had used gradle before and talked me through her build file and style configurations that I cloned and used to get started.

    Checkstyle and PMD really ensure that your code has the correct format and at times when one of them would pass the other would break, which is quite frustrating. There are a lot of syntax issues that I needed to work on with my Java and I definitely disabled some of the style check suggestions that were too stringent, but slowly I have stared to get used to running tests and setting up my Human vs Human tic tac toe with gradle.

    There have been advantages and disagvantages to using vim and gradle for this iteration (instead of Intellij). I will write up a longer post on this topic after a little more research. So far there seems to be quite good reasons to use both, or at least know how to use both.
