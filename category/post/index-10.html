  1. Implementation Patterns

    05 November 2015

    As my minesweeper program is getting bigger and I am starting to think about the different classes that could emerge from the existing ones it already has, it is becoming increasingly important to think about how they should be separated.

    I am currently reading Kent Beck’s Implementation Patterns and he describes the various categories of class separation and state. There are a lot of categories, and I think the only way to really fully understand their use is to actually use them in my code.

    The ones I have come across already are Interfaces, and Superclasses to some extent. He also talks about specialization and the types of questions you should be asking if you want to think about how to separate out classes. Some patterns I have not really used yet are the Implementor pattern and the Inner Class.

    Most of the state based patterns are concerned with how an object changes state. This is primarily done through the internals of the object, so all the methods it contains.

    I am going to try and notice when I use these patterns in my code and write up how I am trying to use them.

  2. Java IO Streams

    05 November 2015

    Java Input Output streams abstract different ways in which you can input and output data. The stream can be sent to or taken from any source, it does not matter. It is really a stream or flow of bytes from one destination to another.

    Java divides streams into bytes or character based categories. The character unit of a character based stream is a Unicode or ASCII character and transfers input and output of 8-bit bytes at a time.

    This is in contrast to FileReader and FileWriter, which even though they internally use FileInputStream and FileOutputStream, read and write two bytes at a time. This can slow down your program and so it can be an advantage to use FileInputStream and FileOutputStream directly to transfer the data across sources.

  4. Functional Programming

    06 November 2015

    I thought I knew what functional programming was but I was wrong! I suppose I need to take one step at a time and I am sure during my apprenticeship I will eventually come to learn a functional language, but there is no harm in knowing the basics. So I thought I would share what I came across.

    A functional language is a language that avoid mutating or changing data, so you would just create new data structures instead of changing ones that already exist. This really helps break a problem down into its component parts.

    Functional programs are also required to be stateless, that is, they should perform every task as if they were performing it for the first time. This is a pretty different way of thinking and I have just got my head around object oriented (OO) programming so am glad I will tackle functional programming once I have a bit more practice at writing OO languages.

  5. File Copy Kata

    09 November 2015

    Last week, I worked on a file copy kata that transferred a file from one location to another. Looking at how bytes are transferred across is really interesting. I am enjoying looking at how hardware interelates with software, because before understanding this, it really seems like magic. It is amazing how you can use something so much and depend on it and not really know how it works. This of course isn’t unique to programming. I am sure most people do not have an indepth understanding of the human body or plumbing systems in their house, but I am enjoying looking at low level concepts, mainly because they are relatively new to me.

    The file copy kata works by, taking a file path as a string and translating that string into a file format. This is then converted into an InputStream that has methods that allow it to read a stream of bytes from the stream of data.

    There are two methods that can read a stream of bytes, there is .read() with no arguements that reads one byte at a time and .read(byte[]) that takes a byte array. There is also .read(byte[] b, int off, int len) that allaw a certain length of bytes to be read from an input stream.

    In my code I used the .read that takes a byte array so I could specify how many bytes it should read across as an int every time it reads and writes the bytes across to the new file location.

    Passing the byte array into String(byte[]) allows for the bytes to be translated in my case into their ASCII representation. This is to do with how Vim encodes the characters, and I will write another blog post about encoding after a bit more research.

    Writing to the output stream effectively does the inverse of reading from the input stream and my transfer method combones the read and write functions into a loop that will read and write a certain amount of bytes from the original file in increments, until it reaches the end of the file at -1.

    I originally forgot to close the stream, and even though this is a small enough program to ignore it. It is always good practice to do so.

    Here is the code: 

    
public class FileCopyKata {
    ReportProgress currentReport = new ReportProgress();

    public String read(String inputFilePath, int amountOfBytes) throws IOException {
        File inputFile = new File(inputFilePath);
        InputStream input = new FileInputStream(inputFile);
        byte[] buffer = new byte[amountOfBytes];

        input.read(buffer);

        return new String(buffer);
    }

    public void write(String destinationFilePath, byte[] buffer) throws IOException {
        File destinationFile =  new File(destinationFilePath);
        OutputStream output = new FileOutputStream(destinationFile);

        output.write(buffer);
    }

    public void transfer(String inputFilePath, String destinationFilePath, int amountOfBytes) throws IOException {
        File inputFile = new File(inputFilePath);
        InputStream input = new FileInputStream(inputFile);
        OutputStream output = new FileOutputStream(new File(destinationFilePath));
        byte[] buffer = new byte[amountOfBytes];

        int data;
        while ((data = input.read(buffer)) != -1) {
            currentReport.progressReport(data, (int) inputFile.length());
            output.write(buffer);
        }
        input.close();
        System.out.print("Done!");
    }
}
