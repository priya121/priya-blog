  2. Primitive and Object Types in Java

    16 October 2015

    Java is a statically typed language, which means that every variable and every expression has a type that must be known at compile time.

    Types limit the values a variable can hold or an expression can produce, but also help reduce errors in the long run.

    So in Java, data is stored in the form of data types and every variable has a data type.

    The data types in Java are split into two :-

    primitive data types and object types/references

    PRIMITIVE TYPES

    The Java memory model specifies how the Java virtual machine works with the computer memory (RAM).

    The value of a variable stored as a primitive type is typically stored in the memory location assigned to that variable on the thread stack

    There are 8–predefined primitive by the language, here they are in order of increasing sizes and they are limited to those sizes:

    “primitive” means a fundamental component that is used to create other, larger parts

    When you create variable stored as a primitive type instead of creating an object on the heap, Java creates a kind of automatic variable that is not a reference. This variable holds the value directly and is placed on the stack making access more efficient.

    int x = 3; > this means at the memory location of x, 3 will be stored there

    Primitive types also come in versions that are objects. Just capitalised and these have their own set of methods that can be called on them.

    OBJECT TYPES

    A variable of an object type is also called a reference. This is because the variable doesn’t contain the object, but contains a reference pointing to the object. These are stored in the heap.

    The different object types in java are :

    interface types, class types and array types

    In order to create an object reference you must instantiate an object first so that you can manipulate it. So you don’t manipulate the object directly, but use a reference to the object. The reference to the object is like a remote control to the actual object the TV. So you can access the object by calling certain methods on the reference.

    When you create an object in Java, it is stored on the heap. There is generally no limit to memory size in this area. To create an object on the heap, you would use the new keyword like this:-

    eg. Integer myInteger = new Integer(45);

    MEMORY MODEL

    Primitive types and object types are stored within the Java Memory Model

    A local variable (one not accessible by the class) may be of a primitive type, in which case it is kept on the thread stack.

    A local variable may also be a reference to an object. In that case the reference (the local variable) is stored on the thread stack, but the object itself if stored on the heap.

    An object may contain methods and these methods may contain local variables. These local variables are also stored on the thread stack.

    The member variables of an object however, are stored on the heap along with the object itself. That is true both when the member variable is of a primitive type, and if it is a reference to an object. The heap contains all objects created in your Java application, regardless of what thread created the object.

    AUTOBOXING

    Before Java 5 you had to call a method on an object version of a primitive type in order to get the value

    But with Java 5, Java automatically converts a primitive type like int into corresponding wrapper class object and unwraps an object if a primitive type is required

    eg. ArrayList intList = new ArrayList(); intList.add(1); //autoboxing–primitive to object

    int number = intList.get(0); // unboxing–object to primitive

    Need to be careful, if you try to convert null to a primitive value you will get a NullPointerException

    Integer myInteger = null; int myInt = myInteger;

    Using operators and comparing values, and can also lead to unnecessary memory use.

  3. Minimizing Method Calls

    15 October 2015

    I do not actively think about the memory that a program takes up when it runs. This is something that I am aware of, but my main concern is that the code works. I am starting to see that unnecessary calls lead to slow running programs and this is probably a factor that I should start considering. The simplest way to do this is to reduce the method calls a class makes if at all possible. In my minesweeper game I had 3 calls to get input from the user when this could have been saved as a single variable and reused. So this is something I will try and keep in mind as I start extracting out classes and trying to speed up the code that I am writing.

  4. Extracting Classes

    14 October 2015

    I have tried to not extract classes out to early when writing a program, even when it is a very small program. However, I may fall into the trap of not extracting them out even when I need to. A sign that a class should be abstracted is that there is a variable being constructed in the class that should really only be a concern of a specific method. This is how the Single Responsibility Principle starts to apply and the question that need to be asked–what is this class doing? comes in handy.
